01 · 프로젝트 개요

추진 배경 및 목적

  • 게임 구매 결정에 리뷰가 큰 영향을 주지만, 인기 게임은 리뷰가 수만 건이라 전체 여론을 파악하기 어렵다.
  • 직접 학습한 딥러닝 감성분류 모델로 리뷰의 긍/부정을 판별하고, LLM으로 장단점을 자연어 요약하며, RAG로 리뷰 근거 기반 질의응답을 제공한다.
  • 동일 데이터에서 비사전학습(LSTM) vs 사전학습(DistilBERT) 모델을 비교해 전이학습의 효과를 정량적으로 검증한다.

핵심 목표

게임 선택(또는 리뷰 직접 입력) → DistilBERT 긍/부정 분류 후:

  • 장/단점 요약 — 긍정·부정 리뷰 그룹에서 각 3개씩 추출
  • 총평 요약 — 긍/부정 비율을 반영한 여론 방향 2~3문장
  • 질문 입력 (Q&A) — 벡터 검색 top-5 리뷰를 근거로 LLM 답변

전처리 ① — 균형 샘플링

왜 균형 샘플링인가 · 원본은 추천 82% : 비추천 18%로 불균형이 심해, 그대로 학습하면 모델이 전부 긍정으로 찍는 다수 클래스 붕괴에 빠진다 (초기 실험에서 실제 발생).

방법 · 클래스별로 셔플 후 각 5천 건씩 뽑아 50:50 균형 샘플 1만 건 구성. 앞부분만 자르면 특정 게임에 편중되므로 반드시 셔플 후 샘플링.

길이 분포 · 짧은 리뷰에 치우친 멱함수형 — 절반 이상이 50단어 미만, 1,000단어+ 극단 장문은 밈 복사붙여넣기가 다수.

균형 test셋의 accuracy는 실제 서비스 분포(82:18)에서의 성능과 다름을 해석 시 유의.

모델 설계 — LSTM 베이스라인 vs DistilBERT

LSTM (비사전학습 베이스라인)

공백 토크나이저→ Embedding 100d→ LSTM 128→ Linear 2

어휘·임베딩을 처음부터 학습. min_freq=2 어휘 구축, PAD=0/UNK=1. 사전지식 없이 이 데이터만으로 학습하는 대조군.

DistilBERT (사전학습 파인튜닝)

WordPiece 토크나이저→ distilbert-base-uncased→ 분류 헤드 2

대규모 코퍼스 사전학습 지식 위에 파인튜닝. HF Trainer + DataCollatorWithPadding, max_len 128.

학습 전략 — 수렴 제어 및 최적 모델 선택

LSTM

파라미터 · Adam(lr 1e-4) · batch 128 · 임베딩 100차원 · hidden 128 · 최대 512토큰
결과 · train loss 0.362 / val loss 0.501 → test accuracy 0.763 · F1 0.753

DistilBERT — 파인튜닝

파라미터 · AdamW(lr 5e-5) · 유효 배치 16(8×누적 2) · 최대 128토큰 · 사전학습 가중치 미세 조정
결과 · 평균 train loss 0.481 / val loss 0.591 → test accuracy 0.856 · F1 0.855

공통 통제: 동일 데이터(균형 1만)·동일 분할(70/15/15)·seed 42 고정·동일 지표(Accuracy/F1).

지표 비교 요약

학습 곡선이 말하는 것

train 1k→5.7k에서 LSTM은 +11.8%p(0.638→0.756) 오르지만 아직 상승 중 — 임베딩부터 새로 배우느라 데이터에 굶주린 상태다. 반면 DistilBERT 참고선(0.856)은 모든 지점 위에 있다. 사전학습 = 수만 건 분량의 데이터를 공짜로 얻는 효과.

RAG ① — 리뷰를 벡터로 저장해 둔다 (색인)

LLM에게 그냥 물으면 학습 데이터에 없는 게임 정보를 지어낼(환각) 수 있다. RAG는 실제 리뷰를 근거로 쥐여주는 구조인데, 그러려면 먼저 리뷰를 검색 가능한 형태로 저장해 둬야 한다.

리뷰 10,810건 → ① 임베딩 (all-MiniLM-L6-v2) → ② Chroma에 저장

① 임베딩 — 문장을 숫자 벡터로

임베딩 모델이 리뷰 한 건을 384개의 숫자(벡터)로 바꾼다. 핵심 성질: 의미가 비슷한 문장일수록 가까운 벡터가 된다 — "so fun"과 "really enjoyable"은 단어가 달라도 벡터로는 이웃이 된다. 이 성질 덕분에 나중에 "의미로 검색"이 가능해진다.

② Chroma 저장 — 벡터 + 원문 + 꼬리표

벡터만 저장하면 검색 후 보여줄 것이 없으므로, Chroma에는 벡터·리뷰 원문·메타데이터(게임 이름, 긍/부정 라벨)를 한 세트로 저장한다. 게임 이름 꼬리표 덕분에 "이 게임 리뷰 중에서만 검색"이 가능하다. 결과물은 89MB 파일로 영속화되어 배포 환경에서 그대로 쓴다.

이 색인 작업은 한 번만 수행한다(노트북 06). 질문할 때마다 1만 건을 다시 임베딩하면 매번 수십 초가 걸리므로, 미리 계산해 저장해 두는 것.

04 · LLM 요약 · RAG Q&A

RAG ② — 질문이 오면 이렇게 답한다 (질의응답)

사용자가 게임을 고르고 질문하면, 저장해 둔 색인에서 근거를 찾아 LLM에게 건네는 과정이 실시간으로 일어난다.

질문 입력 → ① 질문도 벡터로 → ② 벡터 검색 (게임 필터 + 유사 리뷰 5개) → ③ LLM에 근거로 전달 → 답변 + 근거 리뷰

① 질문 임베딩

색인 때와 같은 임베딩 모델로 질문을 벡터로 바꾼다. 같은 공간에 있어야 거리 비교가 성립한다.

② 의미로 검색

선택한 게임의 리뷰 중에서 질문과 의미가 가장 비슷한 리뷰 5개를 고른다(코사인 유사도). 키워드가 달라도 잡힌다 — "buggy?"로 물어도 "crashes constantly"가 검색됨.

③ 근거 기반 생성

찾은 리뷰 5개를 LLM에게 건네며 "이 리뷰들에 있는 내용으로만 답하고, 없으면 모른다고 답하라"고 지시. 답변 옆에 근거 원문도 노출해 검증 가능.

실행 예 — The Witcher 3 선택 후 "is this game worth buying?" → "돈이 아깝지 않다", "10/10" 등 근거 5건을 인용한 한국어 답변. LLM의 역할은 다음 슬라이드에서.

04 · LLM 요약 · RAG Q&A

LLM — 받은 근거를 문장으로 바꿔주는 역할

LLM(Qwen2.5-7B-Instruct)은 스스로 게임을 평가하지 않는다. 앞 단계가 골라준 리뷰들을 읽고, 그 내용을 한국어 문장으로 정리해 주는 역할만 한다. 쓰이는 곳은 두 군데다.

분류기가 나눈 긍정/부정 리뷰 + 비율 → Qwen2.5 → 총평 + 장점 3 + 단점 3
검색으로 찾은 근거 리뷰 5개 (앞 슬라이드) → Qwen2.5 → 질문에 대한 답변

쓰임새 ① 총평·장단점 요약

DistilBERT가 리뷰를 긍정/부정으로 나눠 주면, LLM에게 "긍정 60% / 부정 40%"라는 비율과 함께 두 묶음을 건네며 요약을 부탁한다. 긍정 묶음에서 장점 3개, 부정 묶음에서 단점 3개를 뽑고, 비율을 반영한 총평을 쓴다. 분류기가 재료를 나누고, LLM이 글로 정리하는 분업이다.

쓰임새 ② Q&A 답변

앞 슬라이드에서 검색된 근거 리뷰를 받아 답변을 쓴다 (지시·환각 방지는 RAG ② 참조). 여기서 주목할 점은 같은 LLM이 입력만 달리해 두 기능을 수행한다는 것 — 분류 결과를 받으면 요약이, 검색 결과를 받으면 답변이 된다.

1차 모델이 응답에 실패하면 예비 모델(Llama-3.3-70B)이 자동으로 대신 답한다. 모든 계산은 Hugging Face 서버에서 이루어져 서비스 컴퓨터의 자원을 쓰지 않는다.
